# Contributing to AI Maxxing Documentation

We welcome contributions from community members! These guidelines help us maintain quality and consistency.

## Getting Started

1. **Check existing content** — avoid duplicating guides that already exist
2. **Open a GitHub issue** (optional) — if adding significant new documentation, discuss in an issue first
3. **Fork and branch** — create a feature branch for your changes
4. **Follow our style** — see Style Guide below
5. **Submit a pull request** — with a clear title and description
6. **Respond to feedback** — staff may request revisions before merge

## What to Contribute

### ✅ Welcome Contributions
- **Improvements** — clarify confusing sections, fix typos, improve examples
- **New guides** — document techniques, tools, or workflows
- **Corrections** — point out outdated information or factual errors
- **Translations** — community documentation in other languages
- **References** — links to relevant resources or research

### ❌ Not Accepted Without Approval
- **Policy changes** — moderation guidelines, staff procedures
- **Content removal** — only staff may remove or archive content
- **Rebranding** — changes to community identity or official guidelines

See [SECURITY.md](SECURITY.md) for moderation policy details.

## Style Guide

### Markdown
- Use standard GitHub-flavored Markdown
- Headers: `#` for title, `##` for sections, `###` for subsections
- Code blocks: use triple backticks with language (e.g., ` ```python `)
- Lists: prefer unordered (`-`) for most cases

### Tone
- **Helpful and inclusive** — avoid gatekeeping or judgment
- **Clear and concise** — jargon okay if explained first
- **Non-promotional** — focus on education, not sales
- **Evidence-based** — cite sources when making claims

### Structure
```markdown
# Guide Title

Brief intro (1-2 sentences what this covers)

## Prerequisites
- What reader should know/have before starting

## Core Content
- Main explanation in logical sections
- Use examples and code blocks

## Practical Examples
- Real use cases or walkthroughs

## Common Mistakes
- Pitfalls and how to avoid them

## Further Reading
- Links to related topics in this repo
- External references (official docs, research)
```

## Pull Request Process

1. **Title:** `[category] short description` (e.g., `[guide] add jailbreaking techniques`)
2. **Description:** explain what changed and why
3. **References:** link to any related issues or discussions
4. **Testing:** if applicable, verify links work and examples are accurate

## Before Submitting

- [ ] Read related docs to avoid duplication
- [ ] Spell-check and grammar review
- [ ] Verify links are current and working
- [ ] Code examples run without errors
- [ ] Tone matches our community standards
- [ ] No misinformation or unsafe advice
